1. Practical Performance Analysis


* Explain why the shuffle size for part1.py was so small.
The shuffle size for part1.py was small because reduceByKey incorporates a local pre-aggregation step, known as a "Combiner," on each worker machine before the data is shuffled. Instead of shuffling 300 million raw (movieId, rating) records, the workers shuffle significantly fewer records, each being a summary tuple: (movieId, (Total_Rating, Count)). This dramatically reduces the volume of data that must be sent over the network.
* Explain why the shuffle size for part2.py was enormous (or was going to be enormous before it failed).
The shuffle size for part2.py was enormous because groupByKey shuffles all 300 million raw (movieId, rating) records to the designated reducer task without any intermediate aggregation. Every single rating record for every movie, including all 150 million ratings for the 10 "blockbuster" movies, must be transmitted across the network. The shuffle size is essentially equivalent to the size of the original dataset.
* How does this single metric explain the massive difference in performance and stability?
The Shuffle Size is the primary bottleneck. A small shuffle size (in part1.py) means less network I/O, which is the slowest part of a distributed job, leading to fast completion. An enormous shuffle size (in part2.py) creates a network and disk bottleneck, and crucially, leads directly to a memory bottleneck on the receiving worker. The massive size of the incoming data forces a single worker to hold a huge list of all ratings for a "hot key" movie in memory, causing the inevitable OutOfMemoryError (OOM) and job failure.

2. groupByKey Failure Analysis


   * Did your part2.py job crash?
[Your answer here, e.g., Yes, it crashed with an OutOfMemoryError (OOM).]
   * If it did then using your knowledge of the skewed dataset (with its 10 blockbuster movies), explain precisely what was happening inside the Spark worker's memory that caused this error.
The failure was caused by the "hot key" problem combined with groupByKey's design. The dataset is skewed, with 10 "blockbuster" movies receiving 150 million (50%) of the total ratings. When the groupByKey shuffle occurred, all 150 million ratings for a single blockbuster movie were routed to a single Spark worker machine. This worker machine was tasked with creating a colossal, in-memory Python list to hold all 150 million rating values for that single movie key before it could even begin the aggregation (sum() and len()). This list was far too large to fit in the worker's allocated memory, immediately triggering an OutOfMemoryError (OOM), crashing the task and, consequently, the entire Spark job. This is the definition of a single-node bottleneck.

3. reduceByKey Success Analysis


      * Explain precisely why reduceByKey (Part 1) did not have this problem.
reduceByKey did not fail because it uses a Combiner function on the data before the shuffle. This Combiner performs a partial aggregation on each partition. Instead of creating a massive list of raw ratings, it creates a small, fixed-size accumulator (e.g., (Total_Rating, Count)) for each unique movieId found in that partition. This eliminates the need to hold a huge list of raw values in memory at any point.
      * Walk through how it handles the 150 million ratings for one of the "hot" blockbuster movies in a scalable way (i.e., what happens on each machine before the shuffle, and what happens after).
         1. Before Shuffle (Map/Combine Phase): The 150 million ratings for the hot movie are spread across many partitions and many worker machines. On each worker, for that hot movie, the Combiner iterates through only the local ratings it holds. It accumulates these ratings into a single, compact (Total_Rating, Count) pair. For instance, if 1 million ratings for the movie land on Worker A, Worker A sends only one tiny record: (HotMovieID, (Total_Rating_A, 1,000,000)). The original 1 million ratings never leave the worker and are never stored as a massive list.
         2. After Shuffle (Reduce Phase): Only the small, pre-aggregated records (one from each contributing worker) are shuffled to the final reducer. The reducer receives, say, 100 of these compact summary tuples. It then simply combines these 100 tuples (using the same logic: sum the ratings, sum the counts) to get the final result for the hot movie. No single worker ever has to hold 150 million records in memory.

 Conclusion


The core lesson of this assignment is that local pre-aggregation is mandatory for scalable distributed processing, especially with real-world, skewed data. reduceByKey (or similar "Combiner" patterns like Spark's aggregateByKey or Hadoop's Combiner) is the only way to perform aggregations at scale because it drastically reduces the data shuffled across the network, turning a large volume of raw data into a small set of summary statistics. In contrast, groupByKey forces all data for a single key to one node, inevitably leading to network congestion, disk I/O bottlenecks, and catastrophic OutOfMemory errors when dealing with "hot key" data skew.
